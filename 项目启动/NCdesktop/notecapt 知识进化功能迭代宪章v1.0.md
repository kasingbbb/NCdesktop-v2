## 第一部分：三个核心重构原则

在进入具体实现之前，必须对齐三个原则。它们决定了所有细节的取舍方向。

### 原则一：用户不标记，系统自推断

> **用户零主动操作，系统从信号中还原意图。**

传统知识管理工具的最大失败原因是"标记负担"——系统不断要求用户填写标签、分类、归属。用户在学习状态下没有元认知余量来做这件事。

本系统的设计方向：**用户的每一次上传行为本身就是信号，系统从信号中推断意图，不向用户发问。** 只有当系统判断出现严重歧义（置信度 < 0.4）时，才推送一个最简单的确认，且是二选一，不是开放填写。

### 原则二：知识 > 概念，洞见级 > 词条级

> **系统输出的不是概念列表，而是知识单元——用户真正需要理解和记忆的事情。**

20 篇文章产出 100+ 概念是系统失败，不是成功。正确目标：

```
20 篇文章 → 提取出 ~100 个原始概念（内部状态，用户不直接看）
           → 合成为 10-15 个知识单元（用户的主要交互对象）
           → 概念成为知识单元的内部结构，不直接暴露
```

知识单元是"一件你真正需要理解的事"，而不是"一个你遇到的词"。

### 原则三：渐进式交互，每个屏幕只有一个主动作

> **用户在任何时刻都只需要决定一件事：做还是不做当前这件事。**

UI 设计的核心约束：
- 每个知识单元有明确的"当前状态"和"下一步动作"
- 下一步动作只有一个，且一眼可见
- 用户不需要理解整个系统架构，只需要跟着"下一步"走
- 信息密度随用户深入而增加（不在第一屏展示所有信息）

## 第三部分：核心数据模型重构

### 3.1 知识单元（KnowledgeUnit）—— 取代用户侧"概念"

```typescript
interface KnowledgeUnit {
  id: string;
  libraryId: string;

  // 核心内容
  title: string;                    // "泰勒规则如何描述货币政策反应"（洞见句，非词条）
  coreInsight: string;              // 一句话：这件事的本质是什么
  summary: string | null;           // AI 从文档生成的摘要
  explanation: KnowledgeExplanation | null;  // 理解框架

  // 组成结构
  constituentConceptIds: string[];  // 内部关联的原始概念 IDs（不对用户暴露）
  sourceAssetIds: string[];

  // 理解状态（状态机）
  status: KnowledgeStatus;          // 见下方状态机
  userNote: string | null;          // 用户自己的话
  lastMirrorFeedback: MirrorFeedbackResult | null;

  // 进化追踪
  understandingSnapshots: UnderstandingSnapshot[];
  depthLevel: 1 | 2 | 3 | 4 | 5;

  // 时间
  firstCapturedAt: string;
  lastReviewedAt: string | null;
  nextReviewDue: string | null;     // 间隔复习计算值
  createdAt: string;
  updatedAt: string;
}

// 知识状态机（用户的进度路径）
type KnowledgeStatus =
  | 'raw'          // 素材已采集，尚未合成
  | 'synthesized'  // 已合成摘要，待用户了解
  | 'understood'   // 用户读了摘要+框架
  | 'articulated'  // 用户写了自己的理解
  | 'validated'    // 完成 AI 核对
  | 'consolidated' // 经过至少 1 次间隔复习
  | 'mastered';    // 技能验证通过
```

### 3.2 信号推断结果（AssetInference）

```typescript
interface AssetInference {
  assetId: string;

  // 时间聚类
  sessionId: string | null;         // 推断所属的采集会话（±2小时窗口）
  sessionPeerIds: string[];         // 同一会话的其他素材

  // 内容推断
  dominantTopics: string[];         // 主题关键词
  noveltyScore: number;             // 0=复习已有 / 1=全新内容
  closestKnowledgeIds: string[];    // 最相似的已有知识单元
  closestScores: number[];

  // 分类推断
  inferredCourse: string | null;    // 推断所属课程
  inferredType: AssetInferredType;
  isSupplementary: boolean;         // 是否是对已有内容的补充
  supplementTargetId: string | null;// 补充目标（知识单元 ID）

  // 系统置信度
  confidence: number;               // 0-1，<0.4 时触发最简确认
  ambiguityReason: string | null;   // 置信度低时的原因说明
}

type AssetInferredType =
  | 'class_content'    // 课堂内容
  | 'self_study'       // 自学材料
  | 'reference'        // 参考资料
  | 'review'           // 复习材料（相似内容已存在）
  | 'unknown';
```

### 3.3 语音备注（VoiceMemo）

```typescript
interface VoiceMemo {
  id: string;
  assetId: string;                  // 关联素材（可空：独立语音）
  audioPath: string;
  transcript: string;               // Whisper 转录

  // AI 分类
  memoType: VoiceMemoType;
  linkTargetId: string | null;      // 关联的知识单元 ID
  linkReason: string | null;        // 为什么关联到该知识单元

  capturedAt: string;
}

type VoiceMemoType =
  | 'supplementary'   // 对附近内容的补充说明 → 合并到关联知识单元
  | 'standalone'      // 独立的新思考/记录 → 创建新知识草稿
  | 'question'        // 疑问/不理解 → 标记在相关知识单元上
  | 'connection';     // "这让我想到了..." → 触发跨域连接推断
```

### 3.4 理解快照（UnderstandingSnapshot）— 进化层基础

```typescript
interface UnderstandingSnapshot {
  id: string;
  knowledgeUnitId: string;
  timestamp: string;

  userExplanation: string;
  mirrorCoveredCount: number;
  mirrorCoveredPoints: string[];
  mirrorMissedAreas: string[];
  depthLevelAtTime: 1 | 2 | 3 | 4 | 5;
  sourceAssetCountAtTime: number;   // 当时有多少素材（用于检测后续是否有新增）
}
```

```
主导航（底部 Tab Bar，5项）：

[今天]  [知识库]  [图谱]  [技能]  [设置]
```

每个 Tab 对应飞轮的不同层：

## 第五部分：智能采集引擎设计

### 5.1 信号推断流程

```
新 Asset 上传
     │
     ▼
Step 1: 内容提取（已有）
  rawText + structuredMd + qualityLevel
     │
     ▼
Step 2: 信号采集
  ├── 上传时间戳
  ├── 设备 GPS（如有）
  ├── ±2小时内其他上传的 Assets（时间聚类）
  └── Asset 本身的内容 embedding（向量化）
     │
     ▼
Step 3: 推断计算
  ├── 与现有知识单元计算 cosine similarity
  │     → 最相似知识 + 分数 → 判断 noveltyScore
  ├── 时间聚类 → 是否属于同一个采集会话
  └── 内容主题提取 → 与现有课程标签匹配
     │
     ▼
Step 4: 置信度判断
  ├── confidence > 0.7 → 静默自动归属，用户不感知
  ├── 0.4-0.7 → 轻提示（角落 Toast，可忽略）："已归入宏观经济学"
  └── < 0.4 → 最简确认卡（仅2选1）：
              "这是课堂内容还是自学材料？"
              [课堂] / [自学]
     │
     ▼
Step 5: 触发知识合成管道（后台异步）
```

### 5.2 语音备注采集设计

**触发方式**：应用内悬浮录音按钮（快按开始，松开停止）

**分类逻辑**：

```
录音 → Whisper 转录
     │
     ▼
分类判断（LLM）：

判断依据：
  1. 转录内容的语言结构
     - "这说的是..." / "这里的意思是..." → supplementary（补充说明）
     - "我突然想到..." / "其实这和..." → connection（触发跨域连接）
     - "我不懂..." / "这是什么意思" → question（疑问标记）
     - 独立叙述，与近期内容无明显关联 → standalone（新知识草稿）

  2. 时间上下文
     - 在某次上传后 10 分钟内录制 → 大概率是 supplementary
     - 独立时间录制 → 倾向 standalone

输出：
  supplementary → 自动 merge 到最近关联的知识单元（标记来源）
  standalone    → 创建新知识草稿，进入合成管道
  question      → 在相关知识单元上标记 ❓，进入"待解答"队列
  connection    → 提取双方概念，推送"发现跨域连接"通知
```

**UI 呈现**：处理完成后，在"今天"视图出现一行：`"你的语音已自动归入泰勒规则" [查看]`

## 第六部分：知识合成管道

这是系统的核心智能环节——把原始概念提炼为知识单元。

```
原始概念集合（per library）
     │
     ▼
Step 1: 概念聚类
  对所有原始概念做 embedding + 聚类（DBSCAN 或 K-means）
  目标：把 100 个概念聚成 10-15 个主题群

     │
     ▼
Step 2: 知识单元命名
  对每个主题群，LLM 生成：
  - title：洞见句式（"XXX如何/为什么/是什么"，而非词条）
  - coreInsight：一句话本质
  - constituentConceptIds：群内的所有概念 IDs

     │
     ▼
Step 3: 摘要生成（已有，复用）
  用已有的 knowledge_generate_summary 能力
  但输入从单概念改为"知识单元 + 其下所有概念的来源素材"

     │
     ▼
Step 4: 关系计算
  - 知识单元之间的 upstream/downstream（基于概念的已有关系）
  - 跨库/跨课程的 cross_domain（基于 embedding 相似度）

     │
     ▼
Step 5: 写入数据库
  knowledge_units 表
  知识单元的初始 status = 'synthesized'
  触发"今天"视图推送：有 N 个新知识待了解
```

**合成时机**：
- 新 Asset 完成提取后，后台异步触发（非阻塞）
- 批量导入完成后，显式触发一次
- 用户手动"重新合成"（在知识库视图操作）

## 第七部分：进化引擎设计

### 7.1 间隔复习调度

```typescript
// 基于艾宾浩斯曲线的复习间隔（天数）
const REVIEW_INTERVALS = {
  1: 1,    // 第1次复习后：1天后再复习
  2: 3,    // 第2次：3天后
  3: 7,    // 第3次：7天后
  4: 14,   // 第4次：14天后
  5: 30,   // 第5次：1个月后
  6: 90,   // 第6次：3个月后
};

// 复习质量影响间隔（参考 SM-2 算法）
// 核对得分高 → 间隔延长 × 1.5
// 核对得分低 → 间隔缩短，回到上一级
```

**推送逻辑**：
- 本地 cron（每天 9:00 AM 执行）
- 计算 nextReviewDue ≤ today 的知识单元
- 按"接近遗忘临界点"排序，取前 3 个推送
- 系统通知：可直接跳转对应知识详情

### 7.2 知识再校准触发器

```
新素材加入后，检查：
  对于每个与新素材相关的知识单元：
    → 与现有 summary 的 sourceAssetCount 对比
    → 若有新增素材 → 设置 isStale = true
    → 在知识详情页顶部显示：
      "📄 有 1 份新素材还没被纳入，是否更新？"
      [重新生成摘要]  [暂时忽略]
```

### 7.3 跨域连接发现

```
后台低优先级任务（每周运行一次，或新知识合成后触发）：

对所有 knowledge_units 做 pairwise embedding 比较
  → cosine similarity > 0.75 且属于不同课程 → 候选跨域连接

过滤：
  → 已建立连接的排除
  → 同课程内排除

推送：
  "今天"视图出现 🔗 卡片：
  "你在[X课程]理解的[A知识]
   和[Y课程]里的[B知识]
   可能在说同一件事。"
  [查看连接] [不感兴趣]
```

### Step 1：知识单元数据模型建立
**工期估计**：3-4天
**优先级**：P0（所有后续步骤的基础）

**工作内容**：
1. 创建 `knowledge_units` 表（包含上述完整字段）
2. 创建 `understanding_snapshots` 表

4. 创建 `voice_memos` 表

**注意事项**：

### Step 2：知识合成管道（Synthesis Pipeline）
**工期估计**：5-7天
**优先级**：P0

**工作内容**：
1. 实现概念聚类逻辑（Rust 侧，调用 embedding API）
2. 实现知识单元命名 LLM 调用（洞见句 title + coreInsight）
3. 复用现有摘要生成能力，适配知识单元输入格式
4. 实现合成管道的触发机制（素材提取完成后异步触发）
5. 实现合成进度事件（`knowledge:synthesis:progress`）

**验收标准**：
- 导入 20 篇文章，产出 10-15 个知识单元（而非 100+ 概念）
- 每个知识单元有 title、coreInsight、关联概念 IDs、来源素材

**关键决策**：
- 聚类算法优先尝试 LLM 直接做主题归纳（更准确），embedding 聚类作为 fallback
- title 必须是洞见句（"X如何Y"），不能是词条（"X"）

### Step 3：信号推断引擎（Smart Auto-tagging）
**工期估计**：5-6天
**优先级**：P0

**工作内容**：
1. 实现时间聚类算法（±2小时窗口识别采集会话）
2. 实现内容 embedding + 与现有知识单元的 cosine similarity 计算
3. 实现 noveltyScore 计算（0=已有相似知识，1=全新内容）
4. 实现推断结果写入 `asset_inferences` 表
5. 实现置信度判断逻辑
6. 实现低置信度时的最简确认 UI（二选一 Toast）

**验收标准**：
- 上传与已有知识相关的素材 → 自动归属，用户零操作
- 上传全新内容 → 正确标记为"新内容"，进入合成管道
- 置信度 < 0.4 时出现二选一确认（不是开放填写）

## 第十部分：设计约束（所有 AI 协作者必须遵守）

在参与本项目任何功能迭代时，以下约束不可违反：

| 编号 | 约束 | 原因 |
|------|------|------|
| K1 | 用户看到的主要单元是**知识单元**，不是概念 | 概念粒度太细，用户不会看 |
| K2 | 任何自动标记/归类，**不允许弹出要求用户文字填写的界面** | 降低认知负担 |
| K3 | 每个知识单元的详情页，**必须有且只有一个"推荐下一步"的主操作** | 渐进式交互原则 |
| K4 | 知识单元的 title **必须是洞见句（"X如何/为什么/是什么"）**，不能是词条 | 洞见级 > 词条级 |
| K5 | AI 生成的所有内容**必须有来源标注**（来自哪个素材） | 知识基于用户文档的承诺 |
| K6 | 语音备注**不允许要求用户选择类型**，系统自动分类 | 同 K2 |
| K7 | 跨域连接**必须视觉上区别于课程内关联**（虚线+高亮） | 这是最有价值的信息 |
| K8 | 技能验证**不使用选择题**，只用开放分析题 | 选择题无法验证真实理解 |
| K9 | MCP 对外服务的回答**必须来自用户自己的知识内容**，不能混入通用 LLM 知识 | 认知主权原则 |
| K10 | 所有新功能的实现**必须在现有架构约束（A1-A8）下进行** | 见 KNOWLEDGE_DESIGN_CHARTER.md |

## 附：核心 Prompt 设计方向

### 知识单元命名 Prompt（Step 2）
```
你是一个知识提炼专家。以下是从用户文档中提取的一组概念，它们来自同一主题群。

概念列表：{concept_names}
相关原文片段：{source_excerpts}

请生成一个"知识单元"：
1. title：用一句话说明"这件事的本质/规律/机制"，格式为"[X]如何/为什么/是什么[Y]"，不能是词条名
2. coreInsight：一句话，这件事最核心的洞见是什么
3. 注意：title 应该是用户真正需要理解的事，而不是一个术语

示例好的 title："泰勒规则如何描述央行对通胀的反应函数"
示例不好的 title："泰勒规则"（这是词条，不是洞见）
```

### 概念聚类 Prompt（Step 2）
```
以下是从用户学习材料中提取的 {N} 个概念，请将它们归纳为 {target_count} 个主题群。

原则：
- 每个主题群代表"一件用户需要真正理解的事"
- 数量控制在 10-15 个（即使有 100+ 个原始概念）
- 把微小的细节概念合并到更大的主题下
- 同一主题的不同视角/表述归为一群

输出格式：JSON 数组，每项 {group_name, concept_ids[], reason}
```

### 语音分类 Prompt（Step 4）
```
用户录制了一段语音备注，转录如下：
"{transcript}"

录制时间：{timestamp}
录制前10分钟内用户刚上传了：{recent_assets}（可能为空）

请判断这段语音的类型：
- supplementary：用户在解释/补充近期上传的内容
- standalone：用户在记录一个独立的想法/知识
- question：用户在表达疑问或不理解
- connection："这让我想到了"类，建立跨域联系

同时给出：
- linkTargetId：如果是 supplementary/question，最可能关联的素材或知识单元
- confidence：0-1 的置信度

输出 JSON。
```
