## 二、最大并行方案（3 Agent）

### Agent 分配

```
时间 ──────────────────────────────────────────────────────────────▶

┌─ Agent α（Rust 数据 + 引擎）─────────────────────────────────────┐
│                                                                   │
│  S1 → S2 → S3 → S4 → S5 → S6 → S7 → S8 → S12 → S13 → S14     │

│                                                                   │
└───────────────────────────────────────────────────────────────────┘

┌─ Agent β（前端 UI + 灵魂功能）────────────────────────────────────┐
│                                              ↓ 等待 S4              │
│                     S9 → S10 → S11 → [等 S14] → S15 → S16 → S17 │

### 各 Agent 职责

| Agent | 角色定位 | 负责步骤 | 技术重点 |
|-------|---------|---------|---------|
| **α** | Rust 后端工程师 | S1-S8, S12-S14 | Tauri、Rust、SQLite、音频处理、文件系统 |
| **β** | 前端交互工程师 | S9-S11, S15-S17 | React、Canvas 渲染、动效、时间轴交互 |
| **γ** | 独立功能工程师 | S18-S20 | 悬浮窗、OpenAI API、搜索、构建打包 |

## 三、两 Agent 简化方案

如果只有 2 个 Agent，推荐以下分配：

```
┌─ Agent A（后端 + 引擎 + 独立功能）────────────────────────────────┐
│  S1→S2→S3→S4 → S5→S6→S7→S8 → S12→S13→S14 → S18→S19           │

│              ↓S4  S9→S10→S11 → [等S14] → S15→S16→S17 → S20     │

## 四、并行开发的协同规则

### 4.1 接口契约先行

在 Agent 分叉前（S4 完成后），必须先锁定以下接口：

| 接口 | 文件 | 约定内容 |
|------|------|---------|
| **TypeScript 类型** | `src/types/*.ts` | 所有数据模型的 interface 定义 |
| **IPC 命令签名** | `src/lib/tauri-commands.ts` | 所有 `invoke()` 的命令名、参数、返回值 |
| **Store 接口** | `src/stores/*.ts` | 所有 Zustand Store 的 State 和 Action 签名 |
| **Tauri Event 名称** | 文档约定 | `llm-stream-chunk`、`sync-progress` 等事件命名 |

> ⚠️ **规则**：接口定义文件由 Agent α 负责创建，Agent β/γ 消费。修改接口需要通知所有 Agent。

### 4.2 分支策略

```
main ─────────────────────────────────────────────────────▶
  │
  ├── dev/agent-alpha ──── S5→S6→S7→S8→S12→S13→S14 ──── PR
  │
  ├── dev/agent-beta ───── S9→S10→S11→S15→S16→S17 ────── PR
  │
  └── dev/agent-gamma ──── S18→S19 ─────────────────────── PR
```

- 每个 Agent 在独立分支开发
- 每完成一个 Step，提交 PR 到 `main`
- 在汇合点（S15 等待 S14）进行 merge 同步

### 4.3 汇合检查点

| 汇合点 | 时机 | 检查内容 |
|--------|------|---------|
| **C1** | S4 完成后 | 三个 Agent 同步接口契约，确认类型定义 |
| **C2** | S8 完成后 | Agent γ 获得 IPC 通道能力，开始 S18 |
| **C3** | S11 + S14 完成后 | Agent β 获得波形渲染器，开始 S15（关键合并点） |
| **C4** | S17 + S18 完成后 | 所有功能就位，合并到 main，开始 S19/S20 |

### 4.4 冲突预防

| 高风险文件 | 可能冲突的 Agent | 预防措施 |
|-----------|----------------|---------|
| `src-tauri/src/main.rs` | α + γ | 使用 `mod` 分模块，各 Agent 只改自己的模块注册 |
| `src/App.tsx` | β + γ | 路由分区管理，各功能独立路由 |
| `src/stores/` | α + β | 每个 Store 文件只有一个 Agent 负责 |
| `Cargo.toml` | α + γ | 按依赖分组排列，减少行冲突 |

## 五、效率对比

| 方案 | 总开发时间 | Agent 数 | 并行效率 |
|------|-----------|---------|---------|
| 串行（1 Agent） | ~80 小时 | 1 | 基准 |
| 2 Agent 并行 | ~50 小时 | 2 | 37% 提速 |
| 3 Agent 并行 | ~35 小时 | 3 | 56% 提速 |

> 注：并行效率受汇合等待时间影响，非线性提速。3 Agent 方案的瓶颈在 C3 汇合点（Agent β 需等待 Agent α 完成 S14）。

---

## 六、推荐策略

**如果使用 AI Agent 开发**，推荐 **2 Agent 方案**：

- 减少协同复杂度
- 冲突风险最低
- 汇合点最少（仅 C1 和 C3）
- Agent A 专注 Rust 后端，Agent B 专注 React 前端，职责清晰
